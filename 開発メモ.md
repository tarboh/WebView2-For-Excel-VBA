## 流れ

### フェーズ1. WebView2Environmentの作成

``` VBA
Private Declare PtrSafe Function CreateCoreWebView2EnvironmentWithOptions Lib "WebView2Loader.dll" ( _
    ByVal browserExecutableFolder As LongPtr, _
    ByVal userDataFolder As LongPtr, _
    ByVal additionalBrowserArguments As LongPtr, _
    ByVal environmentCreatedHandler As LongPtr) As Long
```
### この関数を実行するために必要な準備

#### 1. browserExecutableFolder
    0でよい。
#### 2. userDataFolder フォルダパスの指定
``` VBA
    Dim userDataPath As String: userDataPath = "C:\Temp\VBA_WebView2"
    If Dir("C:\Temp", vbDirectory) = "" Then MkDir "C:\Temp"
```
このuserDataPathを`StrPtr(userDataPath)`で渡す。
#### 3. additionalBrowserArguments
    0でよい。
#### 4. ICoreWebView2CreateCoreWebView2EnvironmentCompletedHandler ハンドラオブジェクトの指定
`CreateCoreWebView2EnvironmentWithOptions`によってWebView2.exeが起動する。

WebView2.exeは起動成功時、引数で渡されたオブジェクトの４番目の関数`Invoke`を呼び出す。<br>
その呼び出しを正しく受けるため、VBA内で仕様通りのCOMオブジェクト`ICoreWebView2CreateCoreWebView2EnvironmentCompletedHandler`を用意しなければならない。<br>
このオブジェクトは以下の関数を持っていることになっている。
```
IUnknown
  0.QueryInterface
  1.AddRef
  2.Release
ICoreWebView2CreateCoreWebView2EnvironmentCompletedHandler
  3.Invoke
```

### 第一段階：対応する関数を標準モジュールに定義し、それらの関数ポインタの配列を`CopyMemory`で仮装関数テーブルとしてメモリ上に展開する方式

``` VBA
' Initializes the WebView2 Environment
Private Sub CreateWebView2Environment()

    ' 1. Store function pointers in the vTable array
    vTable(0) = GetAddr(AddressOf Handler_QueryInterface)
    vTable(1) = GetAddr(AddressOf Handler_AddRef)
    vTable(2) = GetAddr(AddressOf Handler_Release)
    vTable(3) = GetAddr(AddressOf Handler_Invoke)

    ' 2. Allocate heap memory for the vTable
    pVTable = GlobalAlloc(GPTR, (UBound(vTable) + 1) * LenB(vTable(0)))
    CopyMemory ByVal pVTable, vTable(0), (UBound(vTable) + 1) * LenB(vTable(0))

    ' 3. Allocate heap memory for the Object instance itself (points to pVTable)
    pObject = GlobalAlloc(GPTR, LenB(pVTable))
    CopyMemory ByVal pObject, pVTable, LenB(pVTable)

    ' 4. Start creating the WebView2 Environment
    
    ' Prepare a writable user data folder path (Null-terminated Unicode string)
    Dim userDataPath As String
    userDataPath = "C:\Temp\VBA_WebView2" ' Folder will be created automatically if missing

    ' Ensure base directory exists
    If Dir("C:\Temp", vbDirectory) = "" Then MkDir "C:\Temp"

    Dim hr As Long
    hr = CreateCoreWebView2EnvironmentWithOptions(0, StrPtr(userDataPath), 0, pObject)

    If hr <> S_OK Then
        MsgBox "Failed to initialize WebView2 Environment. HRESULT: " & hr & vbCrLf & _
               "Ensure WebView2Loader.dll is in the same directory as this workbook.", vbCritical, "WebView2 Initialization Error"
    End If

    DoEvents

End Sub
```

### 第四段階：標準モジュールのInvoke中継を撤廃し、WebView2から直接クラスのInvokeを呼ばせる
 
第三段階では一旦標準モジュールの`Handler_Invoke`が呼ばれ、その中から`Class1.Handler_Invoke`を中継呼び出ししていた。<br>
第四段階では中継を完全に撤廃し、WebView2Loader.dllから直接`Class1.Handler_Invoke`が呼ばれるようにする。
 
#### 方針
- vTable本体の構築は第三段階と同じ`PointerAccessor`方式を踏襲。
- vTableの4スロットのうち`[0]QueryInterface` / `[1]AddRef` / `[2]Release`は標準モジュールの`AddressOf`をそのまま入れる。
- `[3]Invoke`スロットだけ、`AddressOf`ではなく`VirtualAlloc(PAGE_EXECUTE_READWRITE)`で確保した実行可能領域上の**サンク**のアドレスを入れる。
- サンクは48バイトのx64マシンコードで、COMから`(this, errorCode, pEnvironment)`の3引数で呼ばれた時、引数を1つ右にシフトして`Me`スロットに`ObjPtr(gHandler)`を割り込ませ、`Class1.Handler_Invoke`の実アドレスへ`jmp`する。
- このサンク領域への書き込みだけは`CopyMemory`を使う（VirtualAllocした実行可能領域に対する1回48バイトの書き込みのみなので、Defender対策上の影響は許容できる）。
#### Class1.Handler_Invokeの実アドレスをITypeInfo探索なしで取得する仕掛け
`Class1`に`Handler_Invoke`**1個だけ**を`Public`配置することで、クラスのvTableは以下のレイアウトに固定される。
```
[0] IUnknown::QueryInterface
[1] IUnknown::AddRef
[2] IUnknown::Release
[3] IDispatch::GetTypeInfoCount
[4] IDispatch::GetTypeInfo
[5] IDispatch::GetIDsOfNames
[6] IDispatch::Invoke
[7] Handler_Invoke   ← オフセット 56 (= 7 * 8) で固定
```
よって`*(LongPtr*)(*(LongPtr*)ObjPtr(gHandler) + 56)`で実アドレスが取れる。<br>
通常サンク方式で必要だった`ITypeInfo::GetTypeAttr` / `GetFuncDesc` / `GetDocumentation`を使った関数ポインタキャッシュ構築（`BuildFuncPtrCache`相当）は完全に省略できる。
 
#### 重要な落とし穴：Class1.Handler_Invokeは`Function`ではなく`Sub`で宣言する
当初`Public Function Handler_Invoke(...) As Long`で実装したところ、Invoke本体は正しく呼ばれて`Debug.Print`まで到達するものの、関数を抜けた瞬間にクラッシュした。<br>
原因：VBAの`Function`は内部的に**戻り値書き戻し用のout引数（`Long*`）を末尾に暗黙追加した形**でコンパイルされる。COM側からは引数3個で`__stdcall`呼び出しされてくるのに、VBAクラスメソッド側は引数4個（最後が戻り値書き戻し用ポインタ）として受け取ろうとする。すると、メソッド冒頭でスタックの未定義領域から「戻り値ポインタ」を読み、メソッド末尾でその不正アドレスへ`Handler_Invoke = 0`を書き込み、リターンアドレスや保存レジスタを踏み潰して`ret`時にクラッシュする。<br>
`Sub`に変更すれば暗黙のout引数が消えるため、COM側の`(this, errorCode, pEnvironment)`シグネチャと完全に一致する。<br>
`ICoreWebView2CreateCoreWebView2EnvironmentCompletedHandler::Invoke`の戻り値（HRESULT）はWebView2側が見ていないため`Sub`化しても実害なし。厳密に`S_OK`を返したい場合はサンク末尾を`call → xor eax,eax → ret`構成に変更すれば良い。
 
#### サンクの中身（x64マシンコード）
Win64 calling convention：`RCX=arg0, RDX=arg1, R8=arg2, R9=arg3`
```
COMからの呼び出し時のレジスタ:
    RCX=this, RDX=errorCode, R8=pEnvironment
   ↓
サンクが引数を1つ右にシフトしつつMeスロットに ObjPtr(gHandler) を注入:
    mov r9, r8       ; pEnvironment → arg3
    mov r8, rdx      ; errorCode    → arg2
    mov rdx, rcx     ; this         → arg1
    mov rcx, imm64   ; ObjPtr(gHandler) → arg0 (Meスロット)
    mov rax, imm64   ; Class1.Handler_Invoke の実アドレス
    jmp rax
   ↓
クラス側で受ける:
    Class1.Handler_Invoke(Me=ObjPtr(gHandler), this, errorCode, pEnvironment)
```
レジスタ退避不要なのは、上書き先（`r9`, `r8`, `rdx`）が上書き元（`r8`, `rdx`, `rcx`）の右側にあるため後ろから順に動かせばよく、依存関係が一方向で衝突しないため。

### 第五段階：第四段階のサンクを改修し、クラスのInvokeを`Function`として叩けるようにする
 
第四段階では`Class1.Handler_Invoke`を`Sub`にせざるを得なかった。これは、VBAの`Function`が内部的に末尾に「戻り値書き戻し用out引数(`Long*`)」を暗黙追加した形でコンパイルされるため、サンクが用意した3引数呼び出しでは引数が1個足りずスタックを破壊してしまうのが原因だった。<br>
第五段階ではサンク側でこのout引数を準備して`call`することで、`Function`での宣言を可能にする。これにより、COM側へHRESULTを明示的に返せるようになる（例：`Handler_Invoke = &H80004001` で`E_NOTIMPL`を返す等）。
 
#### 設計の要点
- サンクは`jmp`ではなく**`call`で呼び出して戻ってくる**形に変更。
- 5番目の引数（戻り値ポインタ）はx64 calling conventionにより**スタック上`[rsp+0x20]`**に配置（`[rsp+0x00..0x1F]`はcallee用shadow space）。
- サンク自身が**ローカルスタック領域に戻り値保管場所を確保**し、そのアドレスを5番目引数として渡す。
- メソッドから戻ったらその保管場所から戻り値を読み出して`eax`にロードしてから`ret`する。
#### スタックレイアウト (`sub rsp, 0x38`直後)
```
[rsp+0x00..0x1F]  shadow space (32 bytes)         ← callee が自由に使える領域
[rsp+0x20]        arg4 = pRetVal (戻り値ポインタ)  ← サンクが [rsp+0x28] のアドレスを書く
[rsp+0x28]        retval 保管領域 (8 bytes)        ← Function が戻り値を書き込む先
[rsp+0x30]        alignment padding (8 bytes)
                  ─────────────────────────
                  合計 0x38 = 56 bytes
```
 
#### スタックアラインメント検証
x64では`call`命令直前のRSPは16-byte alignedでなければならない（callが8バイトのリターンアドレスを積んだ結果、callee側エントリ時点でRSPは`16N + 8`になる仕様）。
- WebView2がサンクを`call`した直後：RSPは`16N + 8`
- サンクが`sub rsp, 0x38`(56)した後：RSPは`16N + 8 - 56 = 16N - 48 = 16(N-3)` → **16-byte aligned ✓**
- 内部で`call rax`した直後（callee側エントリ）：`16(N-3) - 8 = 16(N-3) + 8`形式 → **正規の`16N+8` ✓**
#### マシンコード（54バイト）
```
48 83 EC 38              sub  rsp, 0x38          ; ローカル領域確保
4D 89 C1                 mov  r9, r8             ; pEnvironment → arg3
49 89 D0                 mov  r8, rdx            ; errorCode    → arg2
48 89 CA                 mov  rdx, rcx           ; this         → arg1
48 B9 <imm64>            mov  rcx, ObjPtr(gHandler) ; Me注入     → arg0
48 8D 44 24 28           lea  rax, [rsp+0x28]    ; 戻り値保管場所のアドレス
48 89 44 24 20           mov  [rsp+0x20], rax    ; arg4 = 戻り値ポインタ
48 B8 <imm64>            mov  rax, pTargetFunc   ; Class.Handler_Invoke
FF D0                    call rax
8B 44 24 28              mov  eax, [rsp+0x28]    ; 戻り値ロード
48 83 C4 38              add  rsp, 0x38          ; ローカル領域解放
C3                       ret
```
 
レジスタ退避が不要なのは、上書き先(`r9`/`r8`/`rdx`)が上書き元(`r8`/`rdx`/`rcx`)の右側にあるため、後ろから順に動かせば依存関係が一方向で衝突しないため（第四段階と同じ理由）。

``` VBA
    b(i) = &H48: b(i + 1) = &HB9: i = i + 2
    CopyMemory b(i), pSelfObj, 8: i = i + 8
 
    ' lea rax, [rsp+0x28]   (戻り値保管場所のアドレス)
    b(i) = &H48: b(i + 1) = &H8D: b(i + 2) = &H44: b(i + 3) = &H24: b(i + 4) = &H28: i = i + 5
```

``` VBA
' ============================================================
' Invokeサンクの構築 (PointerAccessor方式・CopyMemory完全撤廃)
'
'   マシンコードは第五段階と同一だが、末尾に int3 (0xCC) を 2 バイト
'   パディングして合計 56 bytes (= 8 × 7) に揃える。これにより
'   MemLongPtr の 7 回書き込みだけで実行可能領域への転送が完了する。
'   int3 は ret の後ろなので通常実行では到達しない (万一フォールスルー
'   した場合はデバッガが捕まえる安全策)。
'
'   マシンコード (56 bytes):
'     48 83 EC 38              sub  rsp, 0x38
'     4D 89 C1                 mov  r9, r8
'     49 89 D0                 mov  r8, rdx
'     48 89 CA                 mov  rdx, rcx
'     48 B9 <imm64>            mov  rcx, ObjPtr(gHandler)
'     48 8D 44 24 28           lea  rax, [rsp+0x28]
'     48 89 44 24 20           mov  [rsp+0x20], rax
'     48 B8 <imm64>            mov  rax, pTargetFunc
'     FF D0                    call rax
'     8B 44 24 28              mov  eax, [rsp+0x28]
'     48 83 C4 38              add  rsp, 0x38
'     C3                       ret
'     CC CC                    int3 / int3        ← パディング (実行されない)
' ============================================================
Private Function BuildInvokeThunk( _
    ByVal pSelfObj As LongPtr, _
    ByVal pTargetFunc As LongPtr) As LongPtr

    Const THUNK_SIZE As Long = 56  ' = 8 × 7
    Dim addr As LongPtr
    addr = VirtualAlloc(0, THUNK_SIZE, MEM_COMMIT Or MEM_RESERVE, PAGE_EXECUTE_READWRITE)
    If addr = 0 Then Exit Function

    Dim b(0 To THUNK_SIZE - 1) As Byte, i As Long
    i = 0

    ' sub rsp, 0x38
    b(i) = &H48: b(i + 1) = &H83: b(i + 2) = &HEC: b(i + 3) = &H38: i = i + 4

    ' mov r9, r8
    b(i) = &H4D: b(i + 1) = &H89: b(i + 2) = &HC1: i = i + 3
    ' mov r8, rdx
    b(i) = &H49: b(i + 1) = &H89: b(i + 2) = &HD0: i = i + 3
    ' mov rdx, rcx
    b(i) = &H48: b(i + 1) = &H89: b(i + 2) = &HCA: i = i + 3

    ' mov rcx, imm64   (ObjPtr(gHandler))
    b(i) = &H48: b(i + 1) = &HB9: i = i + 2
    MemLongPtr(VarPtr(b(i))) = pSelfObj: i = i + 8        ' ← CopyMemoryから差し替え

    ' lea rax, [rsp+0x28]
    b(i) = &H48: b(i + 1) = &H8D: b(i + 2) = &H44: b(i + 3) = &H24: b(i + 4) = &H28: i = i + 5
    ' mov [rsp+0x20], rax
    b(i) = &H48: b(i + 1) = &H89: b(i + 2) = &H44: b(i + 3) = &H24: b(i + 4) = &H20: i = i + 5

    ' mov rax, imm64   (pTargetFunc)
    b(i) = &H48: b(i + 1) = &HB8: i = i + 2
    MemLongPtr(VarPtr(b(i))) = pTargetFunc: i = i + 8     ' ← CopyMemoryから差し替え

    ' call rax
    b(i) = &HFF: b(i + 1) = &HD0: i = i + 2

    ' mov eax, [rsp+0x28]
    b(i) = &H8B: b(i + 1) = &H44: b(i + 2) = &H24: b(i + 3) = &H28: i = i + 4

    ' add rsp, 0x38
    b(i) = &H48: b(i + 1) = &H83: b(i + 2) = &HC4: b(i + 3) = &H38: i = i + 4

    ' ret
    b(i) = &HC3: i = i + 1

    ' --- ここまで 54 bytes、残り 2 bytes をパディング ---
    ' int3 (デバッガブレーク。通常実行では到達しない)
    b(i) = &HCC: i = i + 1
    b(i) = &HCC: i = i + 1

    ' --- 実行可能領域への転送 (CopyMemory不使用、MemLongPtrを8バイト単位で7回) ---
    Dim k As Long
    For k = 0 To THUNK_SIZE - 1 Step 8
        MemLongPtr(addr + k) = ReadLongPtrFromBytes(b, k)
    Next k

    BuildInvokeThunk = addr
End Function
```

## 第七段階：サンク領域確保を VirtualAlloc から AddressOf 取得方式に切り替える
 
第六段階までで CopyMemory は完全撤廃したが、サンク領域の確保は依然として `VirtualAlloc(PAGE_EXECUTE_READWRITE)` に依存していた。<br>
第七段階では `MouseScroll.bas` / `LibTimers.bas` の手法に倣い、標準モジュールに定義した**空 Sub のコード領域**にサンクを直接書き込むことで、`VirtualAlloc` / `VirtualFree` の API 宣言ごと撤廃する。
 
#### 仕組み
- VBA ランタイムは各プロシージャに対して 100 バイト前後のコード領域を割り当てている。`End Sub` 相当の短いコードしか入っていなくても領域自体は広く確保されている。`MouseScroll.bas` のコメントによれば最大 91 bytes (x64) まで使用可能との実績。
- 我々のサンクは 56 bytes なので余裕で収まる。
- 標準モジュールに `Private Sub InvokeThunkSlot(): End Sub` を定義し、そのアドレスを `AddressOf` で取得して、PointerAccessor (`MemLongPtr`) でマシンコードを上書きする。
- VBA ランタイムが管理するコード領域なので、解放処理 (`VirtualFree` 相当) は不要。
#### 注意点
- **最適化対策**：空 Sub は `Compile On Demand` / `Background Compile` 設定下で実体が確保されない可能性があるため、サンクを書き込む前に**明示的に 1 度呼び出して Force-Compile** する必要がある (`MouseScroll.bas` と同じ手法)。空 Sub なので呼び出しても実害は無い。
- **`AddressOf` の構文制約**：VBAの`AddressOf`演算子は、関数/Subの実引数位置でしか書けない言語仕様。`addr = AddressOf MySub` のように代入式の右辺に直接書くと構文エラーになる。よって `addr = VBA.Int(AddressOf MySub)` のように何かしらの関数の引数として包む必要がある。`VBA.Int` を使うのは整数を整数のまま素通しできる最も無害な関数だから (MouseScroll 等の定石)。なお x64 で上位 32bit が落ちるのを防ぐ Variant 経由 LongLong 化はあくまで副次効果。
- **書き込み後の直接呼び出し禁止**：上書き後に `InvokeThunkSlot` を直接呼ぶと、サンクが `ICoreWebView2...` の引数前提で動くため未定義動作になる。Force-Compile はサンク書き込みより前に行うこと。
#### 第六段階からの変更点
- `Declare PtrSafe Function VirtualAlloc / VirtualFree` および関連定数 (`MEM_COMMIT` / `MEM_RESERVE` / `MEM_RELEASE` / `PAGE_EXECUTE_READWRITE`) を**全て削除**。
- `BuildInvokeThunk` のサンク領域取得方法を `VirtualAlloc(...)` → `VBA.Int(AddressOf InvokeThunkSlot)` に変更。
- `BuildInvokeThunk` 冒頭で `InvokeThunkSlot` を呼び出して Force-Compile。
- `Private Sub InvokeThunkSlot(): End Sub` を新規追加（サンクの宿主）。
- `FinalizeWebView2Environment` から `VirtualFree` 呼び出しを削除（VBA ランタイム管理領域なので不要）。
- マシンコード生成部分（56 バイトの組み立てとパディング、MemLongPtr 7 回での書き込み）は第六段階と完全同一。

``` VBA
'   サンク領域を、標準モジュールに定義した空 Sub のコード領域に

Public Sub CreateWebView2Environment()

    Dim pHandlerInvoke As LongPtr
    pHandlerInvoke = GetClassMethodAddrAtFixedSlot(gHandler, 7)
    If pHandlerInvoke = 0 Then
        MsgBox "Class4.Handler_Invoke のアドレス取得に失敗", vbCritical
        Exit Sub

    m_pInvokeThunk = BuildInvokeThunk(ObjPtr(gHandler), pHandlerInvoke)
    If m_pInvokeThunk = 0 Then

    If pa.sa.cDims = 0 Then

    pa.arr(1) = GetAddr(AddressOf Handler_AddRef)
    pa.arr(2) = GetAddr(AddressOf Handler_Release)
    pa.arr(3) = m_pInvokeThunk

' ============================================================
' Invokeサンクの構築 (空 Sub のコード領域を流用)
'
'   サンク本体のマシンコードは第六段階と同一 (56 bytes)。
'   違いは書き込み先で、VirtualAlloc で確保した実行可能領域ではなく、
'   標準モジュールの空 Sub (InvokeThunkSlot) のコード領域に直接書く。
'
'   Force-Compile (空 Sub の明示的呼び出し) はサンクに上書きする
'   前に行う必要がある。上書き後にこの Sub を呼び出すと、サンクが
'   ICoreWebView2... の引数前提で実行されてしまうため壊れる。
' ============================================================
Private Function BuildInvokeThunk( _
    ByVal pSelfObj As LongPtr, _
    ByVal pTargetFunc As LongPtr) As LongPtr

    ' Force-Compile：空 Sub を一度呼んでおく (最適化対策)
    InvokeThunkSlot

    ' 空 Sub のコード領域のアドレスを取得
    '   x64 では AddressOf を Long で受けると上位 32bit が落ちるため、
    '   VBA.Int(AddressOf ...) で Variant 経由 LongLong として取り出す。
    Dim addr As LongPtr
    addr = VBA.Int(AddressOf InvokeThunkSlot)
    If addr = 0 Then Exit Function

    ' --- マシンコード生成 (第六段階と同一の 56 bytes) ---
    Const THUNK_SIZE As Long = 56  ' = 8 × 7
    Dim b(0 To THUNK_SIZE - 1) As Byte, i As Long
    i = 0

    ' sub rsp, 0x38
    b(i) = &H48: b(i + 1) = &H83: b(i + 2) = &HEC: b(i + 3) = &H38: i = i + 4

    ' mov r9, r8
    b(i) = &H4D: b(i + 1) = &H89: b(i + 2) = &HC1: i = i + 3
    ' mov r8, rdx
    b(i) = &H49: b(i + 1) = &H89: b(i + 2) = &HD0: i = i + 3
    ' mov rdx, rcx
    b(i) = &H48: b(i + 1) = &H89: b(i + 2) = &HCA: i = i + 3

    ' mov rcx, imm64   (ObjPtr(gHandler))
    b(i) = &H48: b(i + 1) = &HB9: i = i + 2
    MemLongPtr(VarPtr(b(i))) = pSelfObj: i = i + 8

    ' lea rax, [rsp+0x28]
    b(i) = &H48: b(i + 1) = &H8D: b(i + 2) = &H44: b(i + 3) = &H24: b(i + 4) = &H28: i = i + 5
    ' mov [rsp+0x20], rax
    b(i) = &H48: b(i + 1) = &H89: b(i + 2) = &H44: b(i + 3) = &H24: b(i + 4) = &H20: i = i + 5

    ' mov rax, imm64   (pTargetFunc)
    b(i) = &H48: b(i + 1) = &HB8: i = i + 2
    MemLongPtr(VarPtr(b(i))) = pTargetFunc: i = i + 8

    ' call rax
    b(i) = &HFF: b(i + 1) = &HD0: i = i + 2

    ' mov eax, [rsp+0x28]
    b(i) = &H8B: b(i + 1) = &H44: b(i + 2) = &H24: b(i + 3) = &H28: i = i + 4

    ' add rsp, 0x38
    b(i) = &H48: b(i + 1) = &H83: b(i + 2) = &HC4: b(i + 3) = &H38: i = i + 4

    ' ret
    b(i) = &HC3: i = i + 1

    ' --- パディング (int3) ---
    b(i) = &HCC: i = i + 1
    b(i) = &HCC: i = i + 1

    ' --- 空 Sub のコード領域へ転送 (MemLongPtr 7 回) ---
    Dim k As Long
    For k = 0 To THUNK_SIZE - 1 Step 8
        MemLongPtr(addr + k) = ReadLongPtrFromBytes(b, k)
    Next k

    BuildInvokeThunk = addr
End Function
```

## 第八段階：EntryPoint スタブ経由の二段ジャンプ構造を導入し、EBMode 判定を活用する
 
第七段階までで CopyMemory / VirtualAlloc を撤廃し、空 Sub のコード領域を流用するサンク方式は完成した。<br>
しかし第七段階の方式では vTable から DummyASM 領域 (= サンク) を**直接**呼び出していたため、Excel が Break Mode 中に WebView2 から Invoke が呼ばれると VBA インタプリタが停止状態で叩かれてクラッシュする危険がある。<br>
第八段階では `LibTimers.bas` / `MouseScroll.bas` の手法に倣い、`EntryPoint` スタブと `DummyASM` スタブの**二段ジャンプ構造**を導入する。これにより VBA ランタイムが用意する **EBMode 判定** (Break Mode 中はコールバックをスキップする安全装置) を組み込むことができる。
 
#### 前提知識：AddressOf で得られるアドレスの正体
ユーザーが標準モジュールに記述した `Private Sub Foo(): End Sub` のような空 Sub について、`AddressOf Foo` で取得できるアドレスは **「ユーザー記述の Sub 本体のエントリポイントそのもの」ではなく、「VBA ランタイムが自動付加した前処理コード (スタブ) の先頭アドレス」**である。<br>
このスタブ部分には EBMode 判定や引数調整などのランタイム処理が組み込まれており、x64 では先頭から +37 バイト付近に EBMode 判定が、+55 バイトの位置に**「次に呼ぶ先のアドレス」を保持する 8 バイトの即値領域**が埋め込まれている。<br>
通常はその即値にユーザー Sub 本体 (空 Sub なら実質「即 ret」のアドレス) が書かれているが、**ここを書き換えれば任意のコードに分岐させることができる**。これが LibTimers / MouseScroll の核心的な仕掛けである。
 
#### 二段ジャンプ構造
```
WebView2 が vTable[3] を call
        ↓
EntryPoint スタブ先頭            ← vTable[3] にはこのアドレスを登録
        ↓
EntryPoint+0..+37 の VBA ランタイムプロローグ (EBMode 判定含む)
        ↓
   ┌── EBMode = 2 (Break Mode) なら以降スキップ → 即 ret
   │
   └── 通常時は EntryPoint+55 の即値が指す先へ間接呼び出し
        ↓
DummyASM スタブ領域 (= 我々の 56 バイトサンク)   ← EntryPoint+55 にこのアドレスを書き込む
        ↓
サンクが引数シフト + ObjPtr 注入
        ↓
Class.Handler_Invoke
```
 
`EntryPoint+55` の 8 バイト即値領域は、x64 命令で表現すれば直前に `mov rax, imm64` のオペコード (`48 B8`、計 10 バイト命令) があり、そのうちの 8 バイトの imm64 部分に該当する (LibTimers のコード解析より)。書き換えると、その後に続く `call rax` または `jmp rax` でジャンプ先が変わる。
 
#### 第七段階からの変更点
- 空 Sub を `InvokeThunkSlot` 1 個から **`EntryPoint` + `DummyASM` の 2 個**に増やす。
- サンクの書き込み先を `InvokeThunkSlot` から `DummyASM` に変更。
- vTable[3] に登録するアドレスを「サンクの先頭」から「`AddressOf EntryPoint`」に変更。
- `EntryPoint + 55` (= `LATE_BIND_OFFSET`) の位置に `DummyASM` のアドレスを `MemLongPtr` で書き込む処理を追加。
サンク本体のマシンコード (56 バイト) は第六〜七段階と完全同一。スタックアラインメントの心配があったが、`call` 命令の不変条件 (callee entry 時の RSP は `16N+8` 形式) のおかげで EntryPoint 経由でも問題なく整合した。
 
#### LATE_BIND_OFFSET の値
x64 では **55** で固定 (LibTimers / MouseScroll の実績値)。VBA のバージョンや Office のビット数によってこの位置が変わる可能性が理論上はあるが、現行の VBA7 + x64 環境では 55 で問題なく動作する。

```
' ============================================================
' 二段ジャンプ構造のセットアップ
'
'   (1) DummyASM スタブ領域に 56 バイトのサンクマシンコードを書き込む
'   (2) EntryPoint+55 にある「次に呼ぶ先のアドレス」を DummyASM のアドレスで上書きする
'
'   戻り値: vTable に登録すべきアドレス = AddressOf EntryPoint
' ============================================================
Private Function SetupTwoStageThunk( _
    ByVal pSelfObj As LongPtr, _
    ByVal pTargetFunc As LongPtr) As LongPtr

    ' Force-Compile (両方のスタブを実体化させる)
    EntryPoint
    DummyASM

    Dim pEntry  As LongPtr: pEntry = VBA.Int(AddressOf EntryPoint)
    Dim pDummy  As LongPtr: pDummy = VBA.Int(AddressOf DummyASM)
    If pEntry = 0 Or pDummy = 0 Then Exit Function

    ' (1) DummyASM 領域にサンクを書き込む
    WriteThunkMachineCode pDummy, pSelfObj, pTargetFunc

    ' (2) EntryPoint+55 にある間接ジャンプ先アドレスを DummyASM に差し替える
    MemLongPtr(pEntry + LATE_BIND_OFFSET) = pDummy

    SetupTwoStageThunk = pEntry
End Function


' ============================================================
' サンクのマシンコードを指定アドレスに書き込む (第六?七段階と同一)
'
'   マシンコード (56 bytes):
'     48 83 EC 38              sub  rsp, 0x38
'     4D 89 C1                 mov  r9, r8
'     49 89 D0                 mov  r8, rdx
'     48 89 CA                 mov  rdx, rcx
'     48 B9 <imm64>            mov  rcx, ObjPtr(gHandler)
'     48 8D 44 24 28           lea  rax, [rsp+0x28]
'     48 89 44 24 20           mov  [rsp+0x20], rax
'     48 B8 <imm64>            mov  rax, pTargetFunc
'     FF D0                    call rax
'     8B 44 24 28              mov  eax, [rsp+0x28]
'     48 83 C4 38              add  rsp, 0x38
'     C3                       ret
'     CC CC                    int3 / int3 (パディング)
' ============================================================
Private Sub WriteThunkMachineCode( _
    ByVal addr As LongPtr, _
    ByVal pSelfObj As LongPtr, _
    ByVal pTargetFunc As LongPtr)

    Const THUNK_SIZE As Long = 56
    Dim b(0 To THUNK_SIZE - 1) As Byte, i As Long
    i = 0

    b(i) = &H48: b(i + 1) = &H83: b(i + 2) = &HEC: b(i + 3) = &H38: i = i + 4
    b(i) = &H4D: b(i + 1) = &H89: b(i + 2) = &HC1: i = i + 3
    b(i) = &H49: b(i + 1) = &H89: b(i + 2) = &HD0: i = i + 3
    b(i) = &H48: b(i + 1) = &H89: b(i + 2) = &HCA: i = i + 3

    b(i) = &H48: b(i + 1) = &HB9: i = i + 2
    MemLongPtr(VarPtr(b(i))) = pSelfObj: i = i + 8

    b(i) = &H48: b(i + 1) = &H8D: b(i + 2) = &H44: b(i + 3) = &H24: b(i + 4) = &H28: i = i + 5
    b(i) = &H48: b(i + 1) = &H89: b(i + 2) = &H44: b(i + 3) = &H24: b(i + 4) = &H20: i = i + 5

    b(i) = &H48: b(i + 1) = &HB8: i = i + 2
    MemLongPtr(VarPtr(b(i))) = pTargetFunc: i = i + 8

    b(i) = &HFF: b(i + 1) = &HD0: i = i + 2
    b(i) = &H8B: b(i + 1) = &H44: b(i + 2) = &H24: b(i + 3) = &H28: i = i + 4
    b(i) = &H48: b(i + 1) = &H83: b(i + 2) = &HC4: b(i + 3) = &H38: i = i + 4
    b(i) = &HC3: i = i + 1

    b(i) = &HCC: i = i + 1
    b(i) = &HCC: i = i + 1

    Dim k As Long
    For k = 0 To THUNK_SIZE - 1 Step 8
        MemLongPtr(addr + k) = ReadLongPtrFromBytes(b, k)
    Next k
End Sub
```

## 第 8.5 段階：偵察フェーズ — EntryPoint スタブの実態解析
 
第八段階で二段ジャンプ構造が完成し、第九段階以降ではハンドラを大量生産する仕組みを設計する必要が出てきた。<br>
そこで「ハンドラごとに専用の空 Sub を 2 つ (EntryPoint + DummyASM) ずつ用意する素朴プール方式」だけでなく、「**EntryPoint スタブ自体を `PointerAccessor` で読み出して別の領域にコピーすることで量産できないか**」という発想が浮上した。<br>
この方式が成立するかどうかは、**EntryPoint スタブのバイト列に RIP 相対アドレッシング命令が含まれていないか**が分水嶺となる (RIP 相対命令はコピー先で disp32 がズレてクラッシュするため、relocation 処理が必要になる)。<br>
そのため、第 8.5 段階としてダンプツール `Module8_5.bas` を作成し、`AddressOf EntryPoint` で取れるスタブ領域の中身を実機で偵察した。

```
' ============================================================
' バイトダンプ (16バイト幅)
' ============================================================
Private Sub DumpBytes(ByVal addr As LongPtr, ByVal count As Long)
    Dim i As Long
    For i = 0 To count - 1 Step 16
        Dim line As String
        Dim ascii As String
        line = "+" & PadL(CStr(i), 3) & " : "

        Dim j As Long
        For j = 0 To 15
            If i + j >= count Then Exit For
            Dim b As Byte: b = ReadByte(addr + i + j)
            line = line & PadL(Hex(b), 2, "0") & " "
            If j = 7 Then line = line & " "  ' 中央に空白

            If b >= 32 And b < 127 Then
                ascii = ascii & Chr$(b)
            Else
                ascii = ascii & "."
            End If
        Next j

        Debug.Print line & " | " & ascii
    Next i
End Sub


' ============================================================
' RIP相対アドレッシング命令の粗いスキャン
'   x64 で代表的な RIP-relative 命令のオペコードパターンを探す。
'
'   完全なディスアセンブラではないので、誤検知 / 見逃しは前提。
'   あくまで「該当命令が含まれてそうかどうかの当たりをつける」用途。
'
'   検出パターン (代表的なもの):
'     48 8B 05 ?? ?? ?? ??  : mov rax, [rip+disp32]
'     48 8B 0D ?? ?? ?? ??  : mov rcx, [rip+disp32]
'     48 8B 15 ?? ?? ?? ??  : mov rdx, [rip+disp32]
'     48 8B 1D / 25 / 2D / 35 / 3D : mov r{bx,sp,bp,si,di}, [rip+disp32]
'     48 8D 05/0D/...       : lea r??, [rip+disp32]
'     FF 15 ?? ?? ?? ??     : call qword ptr [rip+disp32]
'     FF 25 ?? ?? ?? ??     : jmp  qword ptr [rip+disp32]
'     E8 ?? ?? ?? ??        : call rel32  ← これも相対！
'     E9 ?? ?? ?? ??        : jmp  rel32
' ============================================================
Private Sub ScanForRipRelative(ByVal addr As LongPtr, ByVal count As Long)
    Dim found As Long: found = 0
    Dim i As Long

    For i = 0 To count - 5
        Dim b0 As Byte: b0 = ReadByte(addr + i)
        Dim b1 As Byte: b1 = ReadByte(addr + i + 1)
        Dim b2 As Byte: b2 = ReadByte(addr + i + 2)

        Dim mnemonic As String: mnemonic = ""

        ' E8 / E9 (call rel32 / jmp rel32)
        If b0 = &HE8 Then
            mnemonic = "call rel32"
        ElseIf b0 = &HE9 Then
            mnemonic = "jmp rel32"

        ' 48 8B XX (mov r64, [rip+disp32]) ? XX の下位 3 bit が rm=101 (=RIP相対)
        ElseIf b0 = &H48 And b1 = &H8B Then
            If (b2 And &HC7) = &H5 Then
                mnemonic = "mov r64, [rip+disp32]"
            End If

        ' 48 8D XX (lea r64, [rip+disp32])
        ElseIf b0 = &H48 And b1 = &H8D Then
            If (b2 And &HC7) = &H5 Then
                mnemonic = "lea r64, [rip+disp32]"
            End If

        ' FF 15 XX (call qword ptr [rip+disp32])
        ElseIf b0 = &HFF And b1 = &H15 Then
            mnemonic = "call [rip+disp32]"

        ' FF 25 XX (jmp qword ptr [rip+disp32])
        ElseIf b0 = &HFF And b1 = &H25 Then
            mnemonic = "jmp [rip+disp32]"
        End If

        If LenB(mnemonic) > 0 Then
            ' disp32 を読み取って ターゲット絶対アドレスを計算
            Dim instLen As Long
            Dim dispOffset As Long
            If b0 = &HE8 Or b0 = &HE9 Then
                instLen = 5: dispOffset = 1
            ElseIf b0 = &HFF Then
                instLen = 6: dispOffset = 2
            Else  ' 48 8B / 48 8D
                instLen = 7: dispOffset = 3
            End If

            Dim disp32 As Long: disp32 = ReadLong(addr + i + dispOffset)
            Dim target As LongPtr: target = (addr + i + instLen) + disp32

            Debug.Print "  +" & PadL(CStr(i), 3) & " : " & mnemonic & _
                       "   disp32=" & disp32 & "  → target=0x" & HexFmt(target, 16)
            found = found + 1
        End If
    Next i

    If found = 0 Then
        Debug.Print "  RIP相対命令は検出されなかった。"
        Debug.Print "  → スタブのコピー量産は理論的に成立しそう。"
    Else
        Debug.Print
        Debug.Print "  ★ " & found & " 個の RIP相対命令を検出。"
        Debug.Print "  → スタブをそのままコピーすると、これらが指す先がズレてクラッシュする。"
        Debug.Print "    コピー時に disp32 を新しい配置に合わせて再計算 (relocation) する必要あり。"
    End If
End Sub


' ============================================================
' 1 byte 読み出し
'   ReadLongPtr で 8 バイト読み、下位 1 バイトをマスクで取り出す。
'   別途 ByteAccessor を用意する手間を回避する。
'   ※ addr の周辺 7 バイトもアクセスされるが、コード領域中での
'     使用を想定しているため終端問題は無視できる。
' ============================================================
Private Function ReadByte(ByVal addr As LongPtr) As Byte
    Dim val As LongPtr
    val = ReadLongPtr(addr)
    ReadByte = CByte(val And &HFF)
End Function

' ============================================================
' 4 bytes (Long) 読み出し
'   バイト 4 個を組み立てる方式で実装 (PointerAccessor の Long 型版を
'   別途用意する手間を省く)
' ============================================================
Private Function ReadLong(ByVal addr As LongPtr) As Long
    Dim b0 As Long: b0 = ReadByte(addr + 0)
    Dim b1 As Long: b1 = ReadByte(addr + 1)
    Dim b2 As Long: b2 = ReadByte(addr + 2)
    Dim b3 As Long: b3 = ReadByte(addr + 3)
    ' 符号付き Long (リトルエンディアン)
    If b3 < &H80 Then
        ReadLong = b0 Or (b1 * &H100&) Or (b2 * &H10000) Or (b3 * &H1000000)
    Else
        ' 負の値: 2の補数表現
        ReadLong = b0 Or (b1 * &H100&) Or (b2 * &H10000) Or _
                   ((b3 - &H100) * &H1000000)
    End If
End Function

' ============================================================
' 8 bytes (LongPtr) 読み出し
'   第六段階以降で動作実績のあるパターン。
' ============================================================
Private Function ReadLongPtr(ByVal addr As LongPtr) As LongPtr
    Dim pa(0 To 0) As PointerAccessor
    With pa(0)
        .sa.cDims = 1
        .sa.cLocks = 1
        .sa.fFeatures = FADF_AUTO Or FADF_FIXEDSIZE
        .sa.cbElements = PtrSize
        .sa.pvData = addr
        .sa.rgsabound0.cElements = 1
        WritePtrNatively pa, VarPtr(.sa)
        ReadLongPtr = .arr(0)
        .sa.rgsabound0.cElements = 0
        .sa.pvData = NullPtr
    End With
End Function


' ============================================================
' SafeArray ポインタを PointerAccessor の内部 arr フィールドに注入する
' ためのヘルパ。第六段階以降で動作実績のあるパターン。
' ============================================================
Private Sub WritePtrNatively(ByRef ptrs() As LONG_PTR, ByVal ptr As LongPtr)
    ptrs(0) = ptr
End Sub


' ============================================================
' フォーマットヘルパ
' ============================================================
Private Function HexFmt(ByVal val As LongPtr, ByVal width As Long) As String
    Dim h As String: h = Hex(val)
    HexFmt = PadL(h, width, "0")
End Function

Private Function PadL(ByVal s As String, ByVal width As Long, _
                      Optional ByVal padChar As String = " ") As String
    ' Optional のデフォルト値が効かないケースの保険
    If LenB(padChar) = 0 Then padChar = " "

    If Len(s) >= width Then
        PadL = s
    Else
        PadL = String$(width - Len(s), Left$(padChar, 1)) & s
    End If
End Function
```

#### 判明した重要事項
 
**1. スタブの実体長は 91 バイト**<br>
+0..+90 までが意味のある命令で、+91 以降は VBA ランタイムが確保した余白バッファ。第六〜七段階で書き込んでいたサンク 56 バイトはこのバッファ内に納まる。
 
**2. RIP 相対命令はゼロ**<br>
スタブの全命令は以下のいずれかで構成されており、コピー時の relocation が一切不要：
- スタックへの相対アクセス (`[rsp+N]`)
- `mov reg, imm64` による絶対アドレスロード
- `call rax` / `jmp rax` による間接呼び出し
- 短距離 `je rel8` 相対ジャンプ (`74 32`, `74 20`) — オフセットがスタブ内部に閉じているためコピーしても正しく動く
ダンプツールの粗い RIP 相対スキャナが「+23 に jmp rel32」を検出していたのは、実際は +20 の `mov rax, imm64` 命令の imm64 部分のバイト列 `E9` を誤検知したものだった。
 
**3. LibTimers の `+55` の意味が完全に判明**<br>
+53 の `48 B8 ...` (`mov rax, imm64`) 命令のうち、**imm64 フィールドの開始位置が +55**。ここに書き込まれている 8 バイトが「Break Mode でなかった場合の次の呼び出し先アドレス」となる。LibTimers / MouseScroll が `Const lateBindOffset = 55` としていたのはこの位置を指している。
 
**4. EBMode 取得関数のアドレス**<br>
`0x00007FFD9E8DD1E4` という典型的な VBA7.dll / VBE7.dll 内部のアドレスを `mov rax, imm64` で直にロードしている。これはプロセス起動ごとに変わる ASLR 対象アドレスだが、各 Sub のスタブが個別に持っているわけではなく、共通の関数を呼んでいるはず。コピーしてもこの imm64 はそのまま有効。
 
**5. Break Mode 時の安全な復帰経路**<br>
+85 の `xor rax, rax` / `ret 0` により、Break Mode (EBMode = 2) では引数復元もせず即 return する設計。これにより VBA インタプリタが停止状態でもクラッシュしない。我々がスタブをコピーして使う場合も、この安全装置はそのまま機能する。
 
**6. スタブの先頭で参照される `EntryPoint - 4` の意味**<br>
+20 の即値 `0x000001DF171AE9B0` は EntryPoint の先頭アドレス (`0x000001DF171AE9B4`) より 4 バイト手前を指している。これは VBA が Sub に紐づけているメタデータ領域 (識別子? state? 不明) を指していると推測される。<br>
この値を使って null チェックしているだけなので、コピー後に同じ値が入っていれば動作は変わらない (どうせ null ではないので EBMode チェックに進む)。**コピー時に書き換える必要は無い**。
 
#### 第九段階に向けて確定した設計の選択肢
 
スタブのコピー量産方式が **relocation 不要で完全に成立する** ことが確定した。これにより、第九段階で取りうるアプローチは以下の 3 つに整理される：
 
- **案 X (素朴プール)**：`Private Sub EntryPointNN(): End Sub` と `Private Sub DummyASMNN(): End Sub` のペアを N 組並べる。確実だがコード行数が多い。
- **案 Y (巨大 Sub 内に DummyASM 領域を切り出し + EntryPoint × N)**：DummyASM 側だけ巨大 Sub から切り出して節約。EntryPoint は依然として N 個必要。
- **案 Z (1 つの巨大 Sub の中に EntryPoint スタブのコピー + サンクを N 組詰め込む)**：第 8.5 段階の発見に基づく新案。1 つの巨大 Sub のコード領域に対して PointerAccessor で 91 バイトのスタブクローンを N 個コピーし、各クローンの +55 を独立に書き換えてそれぞれ別のサンクを呼ばせる。EntryPoint も DummyASM も標準モジュールに大量に並べる必要が無い。最もエレガント。
#### ダンプツール `Module8_5.bas` の補足
- バイトダンプ (16 バイト幅 × 6 行)、LongPtr ビュー (12 エントリ)、+55 即値領域のスポットライト、RIP 相対命令の粗いスキャン、を出力する。
