## 流れ

### フェーズ1. WebView2Environmentの作成

``` VBA
Private Declare PtrSafe Function CreateCoreWebView2EnvironmentWithOptions Lib "WebView2Loader.dll" ( _
    ByVal browserExecutableFolder As LongPtr, _
    ByVal userDataFolder As LongPtr, _
    ByVal additionalBrowserArguments As LongPtr, _
    ByVal environmentCreatedHandler As LongPtr) As Long
```
### この関数を実行するために必要な準備

#### 1. browserExecutableFolder
    0でよい。
#### 2. userDataFolder フォルダパスの指定
``` VBA
    Dim userDataPath As String: userDataPath = "C:\Temp\VBA_WebView2"
    If Dir("C:\Temp", vbDirectory) = "" Then MkDir "C:\Temp"
```
このuserDataPathを`StrPtr(userDataPath)`で渡す。
#### 3. additionalBrowserArguments
    0でよい。
#### 4. ICoreWebView2CreateCoreWebView2EnvironmentCompletedHandler ハンドラオブジェクトの指定
`CreateCoreWebView2EnvironmentWithOptions`によってWebView2.exeが起動する。

WebView2.exeは起動成功時、引数で渡されたオブジェクトの４番目の関数`Invoke`を呼び出す。<br>
その呼び出しを正しく受けるため、VBA内で仕様通りのCOMオブジェクト`ICoreWebView2CreateCoreWebView2EnvironmentCompletedHandler`を用意しなければならない。<br>
このオブジェクトは以下の関数を持っていることになっている。
```
IUnknown
  0.QueryInterface
  1.AddRef
  2.Release
ICoreWebView2CreateCoreWebView2EnvironmentCompletedHandler
  3.Invoke
```

### 第一段階：対応する関数を標準モジュールに定義し、それらの関数ポインタの配列を`CopyMemory`で仮装関数テーブルとしてメモリ上に展開する方式

``` VBA
' Receives the initialization result from WebView2

End Function

' Initializes the WebView2 Environment
Private Sub CreateWebView2Environment()

    ' 1. Store function pointers in the vTable array
    vTable(0) = GetAddr(AddressOf Handler_QueryInterface)
    vTable(1) = GetAddr(AddressOf Handler_AddRef)
    vTable(2) = GetAddr(AddressOf Handler_Release)
    vTable(3) = GetAddr(AddressOf Handler_Invoke)

    ' 2. Allocate heap memory for the vTable
    pVTable = GlobalAlloc(GPTR, (UBound(vTable) + 1) * LenB(vTable(0)))
    CopyMemory ByVal pVTable, vTable(0), (UBound(vTable) + 1) * LenB(vTable(0))

    ' 3. Allocate heap memory for the Object instance itself (points to pVTable)
    pObject = GlobalAlloc(GPTR, LenB(pVTable))
    CopyMemory ByVal pObject, pVTable, LenB(pVTable)

    ' 4. Start creating the WebView2 Environment
    
    ' Prepare a writable user data folder path (Null-terminated Unicode string)
    Dim userDataPath As String
    userDataPath = "C:\Temp\VBA_WebView2" ' Folder will be created automatically if missing

    ' Ensure base directory exists
    If Dir("C:\Temp", vbDirectory) = "" Then MkDir "C:\Temp"

    ' Call API (pass the string pointer to the 2nd argument, and custom object to the 4th)
    Dim hr As Long
    hr = CreateCoreWebView2EnvironmentWithOptions(0, StrPtr(userDataPath), 0, pObject)

    If hr <> S_OK Then
        MsgBox "Failed to initialize WebView2 Environment. HRESULT: " & hr & vbCrLf & _
               "Ensure WebView2Loader.dll is in the same directory as this workbook.", vbCritical, "WebView2 Initialization Error"
    End If

    DoEvents

End Sub
```

### 第四段階：標準モジュールのInvoke中継を撤廃し、WebView2から直接クラスのInvokeを呼ばせる
 
第三段階では一旦標準モジュールの`Handler_Invoke`が呼ばれ、その中から`Class1.Handler_Invoke`を中継呼び出ししていた。<br>
第四段階では中継を完全に撤廃し、WebView2Loader.dllから直接`Class1.Handler_Invoke`が呼ばれるようにする。
 
#### 方針
- vTable本体の構築は第三段階と同じ`PointerAccessor`方式を踏襲。
- vTableの4スロットのうち`[0]QueryInterface` / `[1]AddRef` / `[2]Release`は標準モジュールの`AddressOf`をそのまま入れる。
- `[3]Invoke`スロットだけ、`AddressOf`ではなく`VirtualAlloc(PAGE_EXECUTE_READWRITE)`で確保した実行可能領域上の**サンク**のアドレスを入れる。
- サンクは48バイトのx64マシンコードで、COMから`(this, errorCode, pEnvironment)`の3引数で呼ばれた時、引数を1つ右にシフトして`Me`スロットに`ObjPtr(gHandler)`を割り込ませ、`Class1.Handler_Invoke`の実アドレスへ`jmp`する。
- このサンク領域への書き込みだけは`CopyMemory`を使う（VirtualAllocした実行可能領域に対する1回48バイトの書き込みのみなので、Defender対策上の影響は許容できる）。
#### Class1.Handler_Invokeの実アドレスをITypeInfo探索なしで取得する仕掛け
`Class1`に`Handler_Invoke`**1個だけ**を`Public`配置することで、クラスのvTableは以下のレイアウトに固定される。
```
[0] IUnknown::QueryInterface
[1] IUnknown::AddRef
[2] IUnknown::Release
[3] IDispatch::GetTypeInfoCount
[4] IDispatch::GetTypeInfo
[5] IDispatch::GetIDsOfNames
[6] IDispatch::Invoke
[7] Handler_Invoke   ← オフセット 56 (= 7 * 8) で固定
```
よって`*(LongPtr*)(*(LongPtr*)ObjPtr(gHandler) + 56)`で実アドレスが取れる。<br>
通常サンク方式で必要だった`ITypeInfo::GetTypeAttr` / `GetFuncDesc` / `GetDocumentation`を使った関数ポインタキャッシュ構築（`BuildFuncPtrCache`相当）は完全に省略できる。
 
#### 重要な落とし穴：Class1.Handler_Invokeは`Function`ではなく`Sub`で宣言する
当初`Public Function Handler_Invoke(...) As Long`で実装したところ、Invoke本体は正しく呼ばれて`Debug.Print`まで到達するものの、関数を抜けた瞬間にクラッシュした。<br>
原因：VBAの`Function`は内部的に**戻り値書き戻し用のout引数（`Long*`）を末尾に暗黙追加した形**でコンパイルされる。COM側からは引数3個で`__stdcall`呼び出しされてくるのに、VBAクラスメソッド側は引数4個（最後が戻り値書き戻し用ポインタ）として受け取ろうとする。すると、メソッド冒頭でスタックの未定義領域から「戻り値ポインタ」を読み、メソッド末尾でその不正アドレスへ`Handler_Invoke = 0`を書き込み、リターンアドレスや保存レジスタを踏み潰して`ret`時にクラッシュする。<br>
`Sub`に変更すれば暗黙のout引数が消えるため、COM側の`(this, errorCode, pEnvironment)`シグネチャと完全に一致する。<br>
`ICoreWebView2CreateCoreWebView2EnvironmentCompletedHandler::Invoke`の戻り値（HRESULT）はWebView2側が見ていないため`Sub`化しても実害なし。厳密に`S_OK`を返したい場合はサンク末尾を`call → xor eax,eax → ret`構成に変更すれば良い。
 
#### サンクの中身（x64マシンコード）
Win64 calling convention：`RCX=arg0, RDX=arg1, R8=arg2, R9=arg3`
```
COMからの呼び出し時のレジスタ:
    RCX=this, RDX=errorCode, R8=pEnvironment
   ↓
サンクが引数を1つ右にシフトしつつMeスロットに ObjPtr(gHandler) を注入:
    mov r9, r8       ; pEnvironment → arg3
    mov r8, rdx      ; errorCode    → arg2
    mov rdx, rcx     ; this         → arg1
    mov rcx, imm64   ; ObjPtr(gHandler) → arg0 (Meスロット)
    mov rax, imm64   ; Class1.Handler_Invoke の実アドレス
    jmp rax
   ↓
クラス側で受ける:
    Class1.Handler_Invoke(Me=ObjPtr(gHandler), this, errorCode, pEnvironment)
```
レジスタ退避不要なのは、上書き先（`r9`, `r8`, `rdx`）が上書き元（`r8`, `rdx`, `rcx`）の右側にあるため後ろから順に動かせばよく、依存関係が一方向で衝突しないため。

``` VBA
' ============================================================
' Invokeサンクの構築 (CopyMemory方式)
' ============================================================
Private Function BuildInvokeThunk( _
    ByVal pSelfObj As LongPtr, _
    ByVal pTargetFunc As LongPtr) As LongPtr
 
    Dim addr As LongPtr
    addr = VirtualAlloc(0, 48, MEM_COMMIT Or MEM_RESERVE, PAGE_EXECUTE_READWRITE)
    If addr = 0 Then Exit Function
 
    Dim b(47) As Byte, i As Long
    i = 0
 
    ' mov r9, r8       (pEnvironment → arg3)
    b(i) = &H4D: b(i + 1) = &H89: b(i + 2) = &HC1: i = i + 3
    ' mov r8, rdx      (errorCode    → arg2)
    b(i) = &H49: b(i + 1) = &H89: b(i + 2) = &HD0: i = i + 3
    ' mov rdx, rcx     (this         → arg1)
    b(i) = &H48: b(i + 1) = &H89: b(i + 2) = &HCA: i = i + 3
    ' mov rcx, imm64   (ObjPtr(gHandler) → arg0 = Me スロット)
    b(i) = &H48: b(i + 1) = &HB9: i = i + 2
    CopyMemory b(i), pSelfObj, 8: i = i + 8
    ' mov rax, imm64   (Class1.Handler_Invoke の実アドレス)
    b(i) = &H48: b(i + 1) = &HB8: i = i + 2
    CopyMemory b(i), pTargetFunc, 8: i = i + 8
    ' jmp rax
    b(i) = &HFF: b(i + 1) = &HE0: i = i + 2
 
    ' バイト列を実行可能領域にコピー
    CopyMemory ByVal addr, b(0), CLngPtr(i)
 
    BuildInvokeThunk = addr
End Function

' クラスのvTableから固定スロットの関数アドレスを取得
'   Class1.Handler_Invoke は Public メソッド1個構成なので slot 7 で確定
' ============================================================
Private Function GetClassMethodAddrAtFixedSlot( _
    ByVal cls As Object, ByVal slotIndex As Long) As LongPtr
 
    If cls Is Nothing Then Exit Function

    If pObj = 0 Then Exit Function

    pVTable = ReadLongPtr(pObj)
    If pVTable = 0 Then Exit Function
 
    GetClassMethodAddrAtFixedSlot = ReadLongPtr(pVTable + slotIndex * PtrSize)
End Function
```

#### 第四段階で残ったCopyMemoryの用途
`BuildInvokeThunk`内のサンク48バイト書き込み1箇所のみ。<br>
将来PointerAccessorを使ってサンク領域へバイト単位で書き込む方式（`pa.sa.cbElements = 1`のByte型SafeArrayでpvDataをサンク領域に向ける）に置き換えれば、CopyMemoryを完全撤廃できる。


### 第五段階：第四段階のサンクを改修し、クラスのInvokeを`Function`として叩けるようにする
 
第四段階では`Class1.Handler_Invoke`を`Sub`にせざるを得なかった。これは、VBAの`Function`が内部的に末尾に「戻り値書き戻し用out引数(`Long*`)」を暗黙追加した形でコンパイルされるため、サンクが用意した3引数呼び出しでは引数が1個足りずスタックを破壊してしまうのが原因だった。<br>
第五段階ではサンク側でこのout引数を準備して`call`することで、`Function`での宣言を可能にする。これにより、COM側へHRESULTを明示的に返せるようになる（例：`Handler_Invoke = &H80004001` で`E_NOTIMPL`を返す等）。
 
#### 設計の要点
- サンクは`jmp`ではなく**`call`で呼び出して戻ってくる**形に変更。
- 5番目の引数（戻り値ポインタ）はx64 calling conventionにより**スタック上`[rsp+0x20]`**に配置（`[rsp+0x00..0x1F]`はcallee用shadow space）。
- サンク自身が**ローカルスタック領域に戻り値保管場所を確保**し、そのアドレスを5番目引数として渡す。
- メソッドから戻ったらその保管場所から戻り値を読み出して`eax`にロードしてから`ret`する。
#### スタックレイアウト (`sub rsp, 0x38`直後)
```
[rsp+0x00..0x1F]  shadow space (32 bytes)         ← callee が自由に使える領域
[rsp+0x20]        arg4 = pRetVal (戻り値ポインタ)  ← サンクが [rsp+0x28] のアドレスを書く
[rsp+0x28]        retval 保管領域 (8 bytes)        ← Function が戻り値を書き込む先
[rsp+0x30]        alignment padding (8 bytes)
                  ─────────────────────────
                  合計 0x38 = 56 bytes
```
 
#### スタックアラインメント検証
x64では`call`命令直前のRSPは16-byte alignedでなければならない（callが8バイトのリターンアドレスを積んだ結果、callee側エントリ時点でRSPは`16N + 8`になる仕様）。
- WebView2がサンクを`call`した直後：RSPは`16N + 8`
- サンクが`sub rsp, 0x38`(56)した後：RSPは`16N + 8 - 56 = 16N - 48 = 16(N-3)` → **16-byte aligned ✓**
- 内部で`call rax`した直後（callee側エントリ）：`16(N-3) - 8 = 16(N-3) + 8`形式 → **正規の`16N+8` ✓**
#### マシンコード（54バイト）
```
48 83 EC 38              sub  rsp, 0x38          ; ローカル領域確保
4D 89 C1                 mov  r9, r8             ; pEnvironment → arg3
49 89 D0                 mov  r8, rdx            ; errorCode    → arg2
48 89 CA                 mov  rdx, rcx           ; this         → arg1
48 B9 <imm64>            mov  rcx, ObjPtr(gHandler) ; Me注入     → arg0
48 8D 44 24 28           lea  rax, [rsp+0x28]    ; 戻り値保管場所のアドレス
48 89 44 24 20           mov  [rsp+0x20], rax    ; arg4 = 戻り値ポインタ
48 B8 <imm64>            mov  rax, pTargetFunc   ; Class.Handler_Invoke
FF D0                    call rax
8B 44 24 28              mov  eax, [rsp+0x28]    ; 戻り値ロード
48 83 C4 38              add  rsp, 0x38          ; ローカル領域解放
C3                       ret
```
 
レジスタ退避が不要なのは、上書き先(`r9`/`r8`/`rdx`)が上書き元(`r8`/`rdx`/`rcx`)の右側にあるため、後ろから順に動かせば依存関係が一方向で衝突しないため（第四段階と同じ理由）。
 
#### 第四段階からの変更点
- 標準モジュール側：**`BuildInvokeThunk`関数のみ全面改修**。それ以外（vTable構築、PointerAccessor、IUnknownスタブ群、エントリ等）は完全同一。
- クラスモジュール側：`Sub Handler_Invoke`を`Function Handler_Invoke ... As Long`に戻し、末尾に`Handler_Invoke = 0`（S_OK）を追加。

``` VBA
' Invokeサンクの構築 (CopyMemory方式・Function対応)
'
'   呼び出し変換：
'     COM側  Invoke(this, errorCode, pEnvironment) → eax = HRESULT
'                ↓
'     クラス  Handler_Invoke(Me, this, errorCode, pEnvironment, pRetVal*)
'             pRetVal* は Function の戻り値書き戻し用 out 引数
' ============================================================
Private Function BuildInvokeThunk( _
    ByVal pSelfObj As LongPtr, _
    ByVal pTargetFunc As LongPtr) As LongPtr
 
    Dim addr As LongPtr
    addr = VirtualAlloc(0, 64, MEM_COMMIT Or MEM_RESERVE, PAGE_EXECUTE_READWRITE)
    If addr = 0 Then Exit Function
 
    Dim b(63) As Byte, i As Long
    i = 0
 
    ' sub rsp, 0x38
    b(i) = &H48: b(i + 1) = &H83: b(i + 2) = &HEC: b(i + 3) = &H38: i = i + 4
 
    ' mov r9, r8       (pEnvironment → arg3)
    b(i) = &H4D: b(i + 1) = &H89: b(i + 2) = &HC1: i = i + 3
    ' mov r8, rdx      (errorCode    → arg2)
    b(i) = &H49: b(i + 1) = &H89: b(i + 2) = &HD0: i = i + 3
    ' mov rdx, rcx     (this         → arg1)
    b(i) = &H48: b(i + 1) = &H89: b(i + 2) = &HCA: i = i + 3
    ' mov rcx, imm64   (ObjPtr(gHandler) → arg0 = Me スロット)
    b(i) = &H48: b(i + 1) = &HB9: i = i + 2
    CopyMemory b(i), pSelfObj, 8: i = i + 8
 
    ' lea rax, [rsp+0x28]   (戻り値保管場所のアドレス)
    b(i) = &H48: b(i + 1) = &H8D: b(i + 2) = &H44: b(i + 3) = &H24: b(i + 4) = &H28: i = i + 5
    ' mov [rsp+0x20], rax   (arg4 = retval ポインタ をスタック引数スロットに配置)

    ' mov eax, [rsp+0x28]   (Function が書き込んだ戻り値をロード)
    b(i) = &H8B: b(i + 1) = &H44: b(i + 2) = &H24: b(i + 3) = &H28: i = i + 4
 
    ' add rsp, 0x38
    b(i) = &H48: b(i + 1) = &H83: b(i + 2) = &HC4: b(i + 3) = &H38: i = i + 4
 
    ' ret
    b(i) = &HC3: i = i + 1
 
    CopyMemory ByVal addr, b(0), CLngPtr(i)
 
    BuildInvokeThunk = addr
End Function
```
